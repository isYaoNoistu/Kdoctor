# Kdoctor V2

## 版本定位

`V2` 是 `Kdoctor` 从“能跑、能查”走向“更稳、更准、更像给值班人看的报告”的收口版本。

如果说 `V1` 解决的是从零到一的问题，那么 `V2` 解决的是三件事：

- 把问题域补得更完整
- 把证据链做得更可信
- 把输出和使用体验收口到可长期内部使用

`V2` 不是继续做大，而是让现有能力真正达到封版可用状态。

## 核心特性

### 1. 问题域覆盖从主干扩展到实战层

`V2` 在 `V1` 基础上继续补齐了更贴近真实运维现场的能力：

- 消费组：`CSM-*`
- Producer / Transaction：`PRD-*`、`TXN-*`
- Security：`SEC-*`
- Storage：`STG-*`
- Docker / Host / Logs 深化：`DKR-*`、`HOST-*`、`LOG-*`
- Topic 规划与副本风险：`TOP-006~011`
- Kafka / KRaft 扩展检查：`KFK-*`、`KRF-*`

也就是说，`V2` 已经不只是“会看 metadata 的脚本”，而是能从：

- 网络入口
- Kafka 控制面
- 副本与 ISR
- 客户端链路
- Docker / 宿主机
- 日志证据

几个层面一起收敛问题。

### 2. 结果表达真正面向值班场景

`V2` 的一个核心成果，是默认输出终于稳定下来：

- 默认终端只展开 `CRIT / FAIL / WARN / ERROR`
- `PASS / SKIP` 默认折叠
- 主因判断限制在少量高优先级结论
- 建议动作和主因一一对应
- Markdown 输出更适合留档、工单与审计

这让报告从“技术结果堆叠”变成了“现场可读结果”。

### 3. 证据语义和可信度显著提升

`V2` 重点强化了证据链，而不是单纯堆规则：

- 覆盖摘要改成三态：已启用且已取证、已启用但未取证、未纳入本次运行
- endpoint 证据去重、限流
- 日志模块明确区分“拿到日志”与“日志足够支持判断”
- `TOP-011` 只输出真正命中的 topic
- root cause 不再简单罗列，而是开始做同源问题归并

这也是 `V2` 可信度明显高于 `V1` 的主要原因。

### 4. 默认主链路完成封版收口

`V2` 封版前对默认主链路做了明确收口：

- 移除默认输出中的 JMX / Metrics / JVM / Quota 噪声
- 清理中英混排、乱码、术语不一致问题
- 构建脚本、版本注入、二进制目录统一
- 文档、用户手册、示例 YAML 和实际行为保持一致

从这个阶段开始，`Kdoctor` 的定位已经很明确：

**一个用于内部 Kafka 巡检与排障的单二进制工具，而不是监控平台。**

## 优化与改变方向

### 1. 从“扩能力”转向“减误导”

`V2` 最重要的转向不是新增更多模块，而是压低误导成本。

这一阶段明确做了这些收口：

- 不再默认展示无法启用的指标类检查
- 只在输入条件成立时才注册相关检查，减少大面积 `SKIP`
- 把上下文提示和直接风险区分开
- 把重复 evidence 和长 evidence 压缩到可读范围内

### 2. 从“检查很多”转向“结论更短”

`V2` 的报告设计理念是：

- 10 秒内看懂有没有真故障
- 30 秒内知道先查哪一层
- 需要深挖时再看 Markdown / JSON 细节

所以 `V2` 的优化重点包括：

- 控制重点问题数量
- 控制单检查 evidence 条数
- 控制 root cause 数量和建议动作数量

### 3. 从“工具视角”转向“现场视角”

`V2` 还做了一类非常重要的优化：按执行视角解释结果。

例如：

- 外部探测看到的 broker 地址，不一定等于 controller quorum 地址
- 宿主机当前 shell 的限制，不等于 Kafka 容器进程的真实限制
- 日志样本少，应该提示“结论可信度受限”，而不是暗示 Kafka 故障

这类优化让工具更像一个真正理解现场上下文的排障助手。

### 4. 封版后的优化边界被明确写死

`V2` 封版后，后续优化方向被明确限定为“小修小补”，而不是继续扩张：

- 修复误报
- 微调阈值
- 继续精炼中文输出
- 根据真实使用反馈修正文案和证据表达

不再继续把工具往平台化、重监控、重依赖方向推进。

## V2 结论

如果用一句话概括 `V2`，它已经不是“一个可以试用的脚本”，而是：

**一个可以正式放到 Kafka 值班现场使用的内部排障工具。**
