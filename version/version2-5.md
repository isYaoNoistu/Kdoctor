# version2-5

## 阶段定位

这一轮的目标不是继续“扩能力面”，而是把 `version2.md` 里最后剩下的编号位和工程收口点全部补齐，让第二阶段从“已经很强”进入“按文档百分百收口”的状态。

换句话说，这一轮重点做的是：

- 把此前已经具备但尚未单独命名的能力补成显式检查项
- 把根因归并和报告输出一起对齐
- 把示例配置补到第二阶段完整阈值集

## 本轮完成内容

### 1. 补齐最后一批显式检查项

本轮新增并接入：

- `DKR-005` Docker 运行时挂载预期检查
- `MET-003` replica fetcher lag 指标检查
- `MET-005` network idle 指标检查
- `MET-006` request idle 指标检查
- `HOST-007` 宿主机磁盘 / inode 容量检查

这些能力之前并不是完全没有，而是分散在：

- `STG-005`
- `TOP-010`
- `JVM-001`
- `JVM-002`
- `STG-001`

现在已经全部补成 `version2.md` 对应的独立检查 ID，后续做版本追踪、测试对表和文档核对时不会再出现“能力有了，但编号没落”的情况。

### 2. 根因归并已同步吸收这批新检查

这轮同步更新了 `root cause` 归并逻辑：

- `MET-003` 现在会并入 Topic / Replica 健康主因
- `MET-005 / MET-006` 会并入 broker 压力、线程池与 backpressure 主因
- `HOST-007` 和 `DKR-005` 会并入宿主机 / Docker / storage 环境主因

这意味着第二阶段现在不仅检查项齐了，而且摘要层也知道如何把它们合并成“对值班有用”的结论，而不是简单平铺结果。

### 3. 配置示例已补到 V2 完整阈值集

`kdoctor.example.yaml` 和根目录 `kdoctor.yaml` 现在已经补上：

- `inode_warn_pct`
- `cert_expiry_warn_days`
- `produce_throttle_warn_ms`
- `fetch_throttle_warn_ms`
- `request_latency_warn_ms`
- `purgatory_warn_count`
- `heap_used_warn_pct`
- `gc_pause_warn_ms`

这样第二阶段的 metrics / storage / TLS / quota / JVM 检查项，不再依赖“代码里有默认值但示例配置没体现”的隐式行为。

### 4. 输出本地化继续收口

这轮把 `quota` 模块也纳入了中文模块映射，避免第二阶段新增的 `QTA-*` 结果在中文报告里仍然显示英文模块名。

## 验证情况

结果通过。

另外，这一轮还专门重新对照了 `version2.md` 中的剩余编号位，确认此前缺口最大的几项已经全部落到代码里。

## 当前判断

到这一轮为止，第二阶段已经不再是“主干基本完成”，而是达到了：

- 检查域覆盖收口
- 版本编号收口
- 根因归并收口
- 示例配置收口

也就是说，`version2.md` 对应的实现目标已经完成到可以认定为“百分百收口”的程度。

## 仍需说明的边界

“百分百收口”指的是：

- 代码实现对齐 `version2.md`
- 编号与规则体系对齐
- 配置与摘要输出对齐

但这不等于：

- 每一个检查域都已经在真实生产环境里逐条做过联调
- 每一个 JMX / TLS / transaction / quota 场景都已经有现网样本

所以第二阶段现在的状态更准确地说是：

- 设计目标已全部落地
- 工程实现已闭环
- 下一步进入真实场景回归和第三阶段演进最合适
