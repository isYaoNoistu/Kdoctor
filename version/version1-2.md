# Version 1-2

## 文档定位

本文档记录的是对 `Version 1` 阶段代码的阶段性评估和下一步优化方向。它不是功能清单，而是对“当前项目离真正可用还差什么”的复盘。

## 综合判断

对当前阶段的结论是：

- 功能骨架完成度：高
- 诊断闭环完成度：中
- 工程化与产品化完成度：偏中下

换句话说，项目已经不是一个 demo，而是一个能跑、能采集、能出基本结论的 Kafka 运维诊断器骨架；但距离“拿起来就能在现场判主因”的成熟工具，还差一层关联诊断能力和一层工程化收口。

## 已经做对的事情

### 1. 产品方向是对的

项目做的不是“只看 Kafka metadata 的小脚本”，而是面向运维排障的诊断工具。它试图区分网络、`advertised.listeners`、KRaft、Topic/ISR、客户端链路、宿主机、容器和日志层，这个方向是对的。

### 2. 代码分层是成型的

`collector`、`checks`、`snapshot`、`runner`、`diagnose`、`output`、`transport` 的结构已经形成，这意味着后续扩展不会被早期结构拖死。

### 3. 规则已经具备明显的运维场景感

例如：

- `NET-002` 已经开始区分视角问题，而不是机械报错
- `KFK-003` 能把问题较准确地指向 `advertised.listeners`

这说明规则不是停留在“能查”，而是在朝“会查”发展。

### 4. 输入面已经具备适配性

项目已经支持：

- `bootstrap-only`
- `profile` 增强
- `compose` 增强

这说明工具没有被某一种部署方式绑死。

### 5. 依赖比较克制

项目依赖规模不大，适合作为 Go 二进制工具持续交付。

## 当前阶段的主要缺口

### 1. 最缺的不是采集，而是诊断层

当时最明显的问题是：

- 已经能发现很多症状
- 但还不够会把症状归并成一个主因

也就是说，检查器比较完整，但诊断器还不够成熟。

### 2. V1 验收项还没有完全闭环

当时缺少比较成体系的：

- healthy / fault 场景样例
- golden 输出
- 更清晰的 schema / 输出资产沉淀

### 3. 调度层偏薄

调度更多还是“并发跑任务”，还不是“具备更细粒度 timeout、隔离和 degrade 策略的执行框架”。

### 4. 输出层还没有完全产品化

当时 Markdown 输出目录已经存在，但 CLI 还没有正式接通到主流程。

### 5. 产品边界和发布形态还需要继续收口

项目当时已经具备较强潜力，但对外呈现和内部实现之间还存在一点落差。

## 建议的优先级

### P0：先补诊断归因引擎

优先补的不是更多检查项，而是：

- 最可能主因
- 为什么这么判断
- 这个主因带来了哪些连带症状
- 下一步先查哪几个点

### P1：做场景夹具和 golden 输出

至少沉淀出若干固定场景：

- 完全健康
- bootstrap 不可达
- `advertised.listeners` 配错
- KRaft quorum 异常
- ISR 缩小
- probe 失败但 metadata 正常

### P1：增强调度与 degrade 机制

需要更清楚地告诉用户：

- 哪些检查没跑
- 为什么没跑
- 是超时、权限、依赖不足还是前置失败

### P1：把 Markdown 输出正式接到 CLI

Markdown 不是附加功能，而是现场沟通和结果留档的高价值输出方式。

### P1：收紧 schema、exit code 和 severity 约定

输出一旦接自动化，结构稳定性就非常重要。

### P2：把 host / docker / capacity 做得更扎实

很多 Kafka 看起来像协议问题，最后根因其实在资源和运行环境。

## 阶段结论

这个阶段的核心判断可以归纳成一句话：

`kdoctor` 当时已经是一个很像样的 Kafka 诊断框架 V1，但还不是一个足够老练的 Kafka 故障医生 V1。

后续优化重点应当先把“会判断主因”做出来，再继续扩展更多检查面。
