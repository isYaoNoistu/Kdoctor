# Version 1-3

## 阶段目标

本阶段的目标是把 `Version 1-2` 里提出的首要优化方向先落地一轮，重点解决三件事：

1. 让诊断层更像“归因器”，而不只是“问题汇总器”
2. 把 Markdown 输出正式接入主流程
3. 把默认终端输出切到中文，降低实际使用门槛

## 本阶段完成的核心工作

### 1. 强化诊断归因层

本轮重点增强了 `diagnose` 层，核心变化包括：

- 不再只是浅层摘取前几个失败项
- 开始按优先级归并主因
- 能对以下高价值场景给出更明确的主因判断：
  - `NET-003` 与 `advertised.listeners`
  - `KFK-004` 与内部主题问题
  - `KRF-*` 与 controller / quorum
  - `CLI-*` 与真实 client probe 失败
  - 探针主题或分区不存在导致的 produce 失败

当前输出已经能更明确地区分：

- 网络层主因
- 内部主题主因
- KRaft 主因
- Topic / ISR 主因
- 客户端链路主因
- 宿主机 / Docker 环境主因

### 2. Markdown 输出正式接线

本阶段把原本目录中已存在但未真正接入的 Markdown 输出正式挂到主流程：

- 支持 `--format markdown`
- 支持按输出文件后缀自动推断格式
- Markdown 报告包含：
  - 概览
  - 主因判断
  - 建议动作
  - 检查项明细

这意味着当前结果已经可以直接发群、贴工单和做留档。

### 3. 默认终端输出改为中文

本阶段把终端输出主标签和大部分检查文案切换为中文，包括：

- 模式
- 配置模板
- 总体状态
- 概览
- 主因判断
- 建议动作
- 检查结果

同时补了一轮本地化映射，重点覆盖：

- probe 失败阶段
- 典型错误短语
- 地址、位点、分区、时延类证据字段

### 4. 清理并重写核心文档

本阶段对根目录和 `version` 目录中的核心 Markdown 文档做了统一清理，目标是解决文档乱码和结构不清的问题。

已重写文档：

- [README.md](../README.md)
- [doc.md](../doc.md)
- [architecture.md](../architecture.md)
- [version1.md](./version1.md)
- [version1-2.md](./version1-2.md)

并新增本阶段记录：

- [version1-3.md](./version1-3.md)

### 5. 补充测试覆盖

本轮补了与本次优化直接相关的测试：

- 诊断摘要测试
- Markdown 渲染测试

## 验证情况

### 自动化验证

已执行并通过：

```powershell
go test ./...
```

### 真实环境验证

已使用脱敏后的三节点地址进行验证：

- `192.168.1.1:9192`
- `192.168.1.1:9194`
- `192.168.1.1:9196`

典型验证命令包括：

```powershell
go run ./cmd/kdoctor probe --bootstrap 192.168.1.1:9192
go run ./cmd/kdoctor probe --bootstrap 192.168.1.1:9192 --json
go run ./cmd/kdoctor probe --bootstrap 192.168.1.1:9192 --format markdown --output .\report.md
```

### 当前测试结论

这轮真实测试里，以下能力已能稳定给出结果：

- `NET-001`、`NET-002`、`NET-003`
- `KFK-001`、`KFK-002`、`KFK-003`
- `KRF-002`
- `TOP-003`、`TOP-004`、`TOP-005`
- `CLI-001`

同时也发现了环境中的真实问题：

- `CLI-002~005` 仍存在失败
- `KFK-004` 暴露出内部主题问题
- 当前失败更像是“主题 / 分区 / 内部主题状态异常”，而不是单纯网络不通

与前一阶段相比，这轮最重要的改进不是“检查项变多”，而是“输出已经更能解释为什么失败”。

## 当前阶段结论

本阶段完成后，`kdoctor` 的状态可以描述为：

- `V1` 主干能力仍然完整
- 诊断层开始具备初步归因能力
- Markdown 输出已经进入正式可用状态
- 默认终端输出已经切换到中文
- 核心文档已经清理并恢复为可维护状态

## 下一阶段建议

在你做完这一轮初步使用测试后，下一阶段建议优先看真实反馈再推进。技术上最值得继续做的是：

1. 根据现场输出继续收敛根因规则
2. 补更多 golden case
3. 继续增强错误短语中文化
4. 细化调度层 timeout / degrade 机制
5. 推进容量与趋势判断
